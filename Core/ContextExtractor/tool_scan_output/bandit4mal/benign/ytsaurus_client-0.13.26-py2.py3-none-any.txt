14	-----------------------------
15	dill: serialize all of python
16	-----------------------------
17	
18	About Dill
19	==========
20	
21	``dill`` extends python's ``pickle`` module for serializing and de-serializing
22	python objects to the majority of the built-in python types. Serialization
23	is the process of converting an object to a byte stream, and the inverse
24	of which is converting a byte stream back to a python object hierarchy.
25	
26	``dill`` provides the user the same interface as the ``pickle`` module, and
27	also includes some additional features. In addition to pickling python
28	objects, ``dill`` provides the ability to save the state of an interpreter
29	session in a single command.  Hence, it would be feasable to save an
30	interpreter session, close the interpreter, ship the pickled file to
31	another computer, open a new interpreter, unpickle the session and
32	thus continue from the 'saved' state of the original interpreter
33	session.
34	
35	``dill`` can be used to store python objects to a file, but the primary
36	usage is to send python objects across the network as a byte stream.
37	``dill`` is quite flexible, and allows arbitrary user defined classes
38	and functions to be serialized.  Thus ``dill`` is not intended to be
39	secure against erroneously or maliciously constructed data. It is
40	left to the user to decide whether the data they unpickle is from
41	a trustworthy source.
42	
43	``dill`` is part of ``pathos``, a python framework for heterogeneous computing.
44	``dill`` is in active development, so any user feedback, bug reports, comments,
45	or suggestions are highly appreciated.  A list of issues is located at https://github.com/uqfoundation/dill/issues, with a legacy list maintained at https://uqfoundation.github.io/project/pathos/query.
46	
47	
48	Major Features
49	==============
50	
51	``dill`` can pickle the following standard types:
52	
53	    - none, type, bool, int, long, float, complex, str, unicode,
54	    - tuple, list, dict, file, buffer, builtin,
55	    - both old and new style classes,
56	    - instances of old and new style classes,
57	    - set, frozenset, array, functions, exceptions
58	
59	``dill`` can also pickle more 'exotic' standard types:
60	
61	    - functions with yields, nested functions, lambdas,
62	    - cell, method, unboundmethod, module, code, methodwrapper,
63	    - dictproxy, methoddescriptor, getsetdescriptor, memberdescriptor,
64	    - wrapperdescriptor, xrange, slice,
65	    - notimplemented, ellipsis, quit
66	
67	``dill`` cannot yet pickle these standard types:
68	
69	    - frame, generator, traceback
70	
71	``dill`` also provides the capability to:
72	
73	    - save and load python interpreter sessions
74	    - save and extract the source code from functions and classes
75	    - interactively diagnose pickling errors
76	
77	
78	Current Release
79	===============
80	
81	The latest released version of ``dill`` is available from:
82	
83	    https://pypi.org/project/dill
84	
85	``dill`` is distributed under a 3-clause BSD license.

100

128	Basic Usage
129	===========
130	
131	``dill`` is a drop-in replacement for ``pickle``. Existing code can be
132	updated to allow complete pickling using::
133	
134	    >>> import dill as pickle
135	
136	or::
137	
138	    >>> from dill import dumps, loads
139	
140	``dumps`` converts the object to a unique byte string, and ``loads`` performs
141	the inverse operation::
142	
143	    >>> squared = lambda x: x**2
144	    >>> loads(dumps(squared))(3)
145	    9
146	
147	There are a number of options to control serialization which are provided
148	as keyword arguments to several ``dill`` functions:
149	
150	* with *protocol*, the pickle protocol level can be set. This uses the
151	  same value as the ``pickle`` module, *HIGHEST_PROTOCOL* or *DEFAULT_PROTOCOL*.
152	* with *byref=True*, ``dill`` to behave a lot more like pickle with
153	  certain objects (like modules) pickled by reference as opposed to
154	  attempting to pickle the object itself.
155	* with *recurse=True*, objects referred to in the global dictionary are
156	  recursively traced and pickled, instead of the default behavior of
157	  attempting to store the entire global dictionary.
158	* with *fmode*, the contents of the file can be pickled along with the file
159	  handle, which is useful if the object is being sent over the wire to a
160	  remote system which does not have the original file on disk. Options are
161	  *HANDLE_FMODE* for just the handle, *CONTENTS_FMODE* for the file content
162	  and *FILE_FMODE* for content and handle.
163	* with *ignore=False*, objects reconstructed with types defined in the
164	  top-level script environment use the existing type in the environment
165	  rather than a possibly different reconstructed type.
166	
167	The default serialization can also be set globally in *dill.settings*.
168	Thus, we can modify how ``dill`` handles references to the global dictionary
169	locally or globally::
170	
171	    >>> import dill.settings
172	    >>> dumps(absolute) == dumps(absolute, recurse=True)
173	    False
174	    >>> dill.settings['recurse'] = True
175	    >>> dumps(absolute) == dumps(absolute, recurse=True)
176	    True
177	
178	``dill`` also includes source code inspection, as an alternate to pickling::
179	
180	    >>> import dill.source
181	    >>> print(dill.source.getsource(squared))
182	    squared = lambda x:x**2
183	
184	To aid in debugging pickling issues, use *dill.detect* which provides
185	tools like pickle tracing::
186	
187	    >>> import dill.detect
188	    >>> dill.detect.trace(True)
189	    >>> f = dumps(squared)
190	    F1: <function <lambda> at 0x108899e18>
191	    F2: <function _create_function at 0x108db7488>
192	    # F2
193	    Co: <code object <lambda> at 0x10866a270, file "<stdin>", line 1>
194	    F2: <function _create_code at 0x108db7510>
195	    # F2
196	    # Co
197	    D1: <dict object at 0x10862b3f0>
198	    # D1
199	    D2: <dict object at 0x108e42ee8>
200	    # D2
201	    # F1
202	    >>> dill.detect.trace(False)
203	
204	With trace, we see how ``dill`` stored the lambda (``F1``) by first storing
205	``_create_function``, the underlying code object (``Co``) and ``_create_code``
206	(which is used to handle code objects), then we handle the reference to
207	the global dict (``D2``).  A ``#`` marks when the object is actually stored.
208	
209
